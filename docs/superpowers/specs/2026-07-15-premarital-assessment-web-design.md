# 婚前关系评估 Web 应用 — 设计规格

日期：2026-07-15  
状态：待用户确认  
来源材料：`婚前关系评估.md`、`婚前关系评估-对照.md`

## 1. 目标

把《婚前关系评估》从 Word/Markdown 手工填写，改成**本地网页**：

1. 双方各自填写（量表 + 开放题）
2. 自动对比差异（一致 / 差 1 分 / 差 ≥2 分）
3. 进度可见、断点续填
4. 数据不出本机（无服务器、无账号）

**成功标准（V1）**

- 两人能在浏览器完成评估填写
- 对照页自动标色，无需手工翻两份文档
- 关闭浏览器再打开，答案仍在
- 可用导出/导入 JSON 在两台设备间合并

**非目标（V1 明确不做）**

- 云同步、登录、多情侣管理
- 七单元全文、辅导者后台
- AI 解读、报告 PDF 美化
- 原生家庭/性生活等其他问卷

## 2. 用户与场景

**用户：** 情侣双方自用（身份标记为「我方 / 对方」，可改显示名）

**场景**

| 场景 | 流程 |
|---|---|
| 同设备 | 切换身份分别填写 → 打开对照页 |
| 不同设备 | A 导出 JSON → 发给 B → B 导入 A 的答卷 → 对照 |

## 3. 信息架构（3 页）

### 3.1 首页 `/`

- 显示双方进度（已答量表数 / 总量，开放题完成数）
- 按钮：继续填写（我方/对方）、查看对照、导出、导入
- 简短说明：数据仅存本机浏览器

### 3.2 填写页 `/fill?person=a|b`

- 左侧/顶部：章节导航（错误一…七，特质一…十）
- 主体：当前章节题目
  - **量表题**：1–5 单选（标签：1 非常同意 … 5 非常不同意）
  - **开放题**：多行文本
- 自动保存（每次作答写入 localStorage）
- 显示本章完成度；上一章/下一章

### 3.3 对照页 `/compare`

- 默认只显示**双方都已作答**的量表题
- 过滤：全部 / 仅差异 / 仅红（差≥2）
- 每题展示：题干、我方分、对方分、色标
- 开放题：并排展示双方文本（不做语义对比）
- 顶部汇总：一致 N、差1 M、差≥2 K、未齐 L

## 4. 题目与数据模型

### 4.1 题库（静态 JSON）

从现有评估模板固化，结构：

```json
{
  "version": 1,
  "scale": { "min": 1, "max": 5, "labels": ["非常同意","同意","中立","不同意","非常不同意"] },
  "chapters": [
    {
      "id": "err1",
      "title": "错误一: 太快决定结婚",
      "part": "errors",
      "items": [
        { "id": "err1_o1", "type": "open", "prompt": "你认识对方多久了？" },
        { "id": "err1_l1", "type": "likert", "prompt": "我和对方已经讨论过彼此的背景。" }
      ]
    }
  ]
}
```

**范围（V1）**

- 第一章错误评估：7 个错误区块（开放题 + 量表）
- 第二章理想特质：10 个特质区块（开放题 + 量表）

**题库基准（锁定）**

- **量表题**：以 `婚前关系评估.md` 与 `婚前关系评估-对照.md` 的**并集**为准，实现时生成 `assessment.json` 并打印最终题数（当前评估 md 可解析约 146 道；若对照 md 多出题干则并入填写页）。
- **开放题**：以 `婚前关系评估.md` 为准（当前约 42 道编号题 / 约 47 个作答块；生成题库时按章节完整收录，允许与 Word 原文微调）。
- 生成脚本须输出：`likert_count`、`open_count`、缺题列表；缺题人工补进 JSON，不静默丢题。

**对照范围：** 仅 `type=likert` 参与色标对比；开放题只并排。

### 4.2 答卷（localStorage）

Key：`premarital.assessment.v1`

```json
{
  "version": 1,
  "people": {
    "a": { "displayName": "我方", "answers": { "err1_l1": 1, "err1_o1": "三年" }, "updatedAt": "ISO" },
    "b": { "displayName": "对方", "answers": {}, "updatedAt": "ISO" }
  }
}
```

**答案空值约定**

- 量表：未作答 = `answers` 中**无该 key**（不要存 `null`）
- 开放题：未作答 = 无该 key；已作答 = 非空字符串（trim 后 length > 0）
- 进度：量表完成 = 有 key 且值为 1–5；开放题完成 = 有 key 且 trim 后非空
- 用户清空已填开放题：删除该 key（视为未完成）

### 4.3 导出文件

单文件 JSON，包含 `people` 全量（或可选只导出当前身份）。

**V1 导入策略（简化）**

1. 校验 `version` 与 `people` 结构，失败则拒绝并提示
2. 弹窗：`将覆盖本地「我方/对方/双方」的全部答案，确认？`（按导入文件里实际包含的 person 提示）
3. 确认后：以导入文件中该 `person` 的 `answers` + `displayName` + `updatedAt` **整体替换**本地对应 person
4. 不做逐题合并、不做按时间戳自动择优

## 5. 对比规则

对双方都有值的量表题：

| 条件 | 标记 | UI |
|---|---|---|
| `a == b` | match | 绿 |
| `abs(a-b) == 1` | near | 黄 |
| `abs(a-b) >= 2` | far | 红 |
| 任一方缺失 | missing | 灰（默认过滤掉） |

三色制（统一，不用“未强调”第四态）：**绿=一致，黄=差1，红=差≥2**；未齐题默认隐藏。

## 6. 技术选型（懒人栈）

| 层 | 选择 | 理由 |
|---|---|---|
| 形态 | 静态单页/多页，无构建亦可 | V1 够用，零部署依赖 |
| 实现 | 原生 HTML + CSS + JS，或极薄 Vite 静态站 | 无后端；本机 `python -m http.server` 可预览 |
| 存储 | `localStorage` + JSON 导入导出 | 符合「仅本机」 |
| 题库 | 仓库内 `data/assessment.json` | 与 UI 分离，后续可加单元 |
| 依赖 | **零 npm 运行时依赖**（可选 dev 用 Vite） | 最短路径 |

推荐目录：

```
web/
  index.html
  fill.html
  compare.html
  css/app.css
  js/
    storage.js
    compare.js
    fill.js
    app.js
  data/assessment.json
```

（若选单页 hash 路由，可收成 `index.html` + 模块；实现阶段二选一，默认多页更直观。）

## 7. UI 原则

- 手机可填：量表大点击区，一题一卡或紧凑列表
- 中文界面
- 不堆教程；关键操作用一行提示
- 对照页默认「仅差异」，减少噪音

## 8. 隐私与安全

- 无网络请求（静态资源除外）；不做分析埋点
- 页面顶部提示：数据在本机，清浏览器会丢；请自行导出备份
- 导出文件含敏感问答，用户自行保管
- 不做密码加密（V1）；若同设备多用户共用电脑，风险自担（可在 README 写明）

## 9. 错误处理

- localStorage 满/禁用：提示并引导仅用导出文件作答（降级：内存编辑 + 强制导出）
- JSON 导入格式错误：拒绝并提示
- 题库缺失：首页报错，不进入填写

## 10. 验证方式

手工检查清单：

1. 我方填 3 道量表 → 刷新仍在
2. 对方填不同分 → 对照色标正确
3. 导出 → 清空 storage → 导入 → 数据恢复
4. 过滤「仅红」只显示差≥2
5. 手机宽度下可点选 1–5

可选：对 `compare.js` 的差值函数做 3 条纯函数断言（无测试框架）。

## 11. 实现分期

**V1（本规格）**

1. 固化 `assessment.json` 题库
2. 存储 + 填写页 + 自动保存
3. 对照页 + 过滤
4. 导入/导出
5. 基础样式与进度

**V1.1（以后）**

- 显示名编辑、打印友好对照
- PWA 离线图标
- 补全开放题题干若题库有缺号

**V2（另开规格）**

- 七单元问卷
- 可选云同步

## 12. 风险与取舍

| 风险 | 取舍 |
|---|---|
| 原 md 题号有抽取缺题 | 量表用两份 md 并集；开放题用评估 md；生成时打印缺题清单 |
| 双方题意不对称（仅一方开放题） | 开放题不强行对比 |
| 清站数据丢失 | 导出备份提示，不做服务端 |

## 13. 已确认决策

1. 主要用户：情侣双方自用  
2. 范围：只做评估 + 双方对照  
3. 数据：仅本机/浏览器本地  
4. 技术方向：静态 Web + localStorage + JSON 导入导出
